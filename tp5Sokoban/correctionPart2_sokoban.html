# Eléments de réponse pour le TP Sokoban - partie II *Disclaimer : les choix d'implémentation sont volontairement naïfs, afin d'illustrer de manière claire la plupart des problèmes liés aux choix de structure de données.* ## En python ! Les choix faits en Prolog restent pertinents : ### actions, positions, états et carte * une action est un couple (verbe, direction), éventuellement associé à un alias d'un caractère via un dictionnaire. On peut définir une classe (mais c'est lourd), ou utiliser `collections.namedtuple` ```python from collections import namedtuple Action = namedtuple('action',('verb','direction')) actions = {d : Action('move',d) for d in 'udrl'} | {d.upper() : Action('move',d) for d in 'udrl'} ``` * les positions sont des couples d'entiers * un état encode la position du personnage et l'ensemble des positions des caisses. On peut utiliser un dictionnaire (*spoiler : ça va poser des problèmes*) ayant pour clés `'me'` et `'boxes'`. La valeur associée à la clé `'boxes'` pourrait être une liste, mais il est plus judicieux d'utiliser un ensemble (*spoiler : ça va aussi poser des problèmes*) de positions car les caisses sont indiscernables ```pyton s0 = {'me': (2, 2), 'boxes': {(4, 4), (3, 4), (6, 5), (6, 1), (6, 4), (2, 3), (6, 3)}} ``` * les caractéristiques non-fluantes au niveau d'une carte, mais qui changent d'une carte à l'autre, peuvent être encodées dans un autre dictionnaire `map_rules` , qui recenserait l'ensembles de positions des cases (ou des murs) ainsi que celui des positions des cibles ```pyton map_rules = {'goals': {(7, 4), (2, 1), (6, 6), (5, 4), (6, 3), (4, 1), (3, 5)}, 'walls': {(4, 0), (4, 3), (3, 1), (4, 6), (5, 7), (8, 0), (0, 2), (8, 3), (0, 5), (8, 6), (1, 0), (1, 6), (7, 7), (4, 2), (3, 0), (5, 0), (5, 6), (3, 6), (8, 2), (8, 5), (1, 2), (0, 4), (7, 0), (6, 7), (3, 2), (5, 2), (8, 4), (8, 1), (8, 7), (1, 1), (0, 3), (2, 0), (0, 6), (2, 6), (6, 0)}, 'actions': actions} ``` ### fonctions intermédiaires * `one_step` joue le rôle du prédicat `positionRelative`. A noter l'utilisation d'un dictionnaire pour implémenter un sélecteur (Rq : il y a depuis la v3.10 un sélecteur natif en python) * `free` permet de créer une couche d'abstraction entre la représentation des données (par exemple, la représentation de la carte à l'aide de cases vides ou bien de murs) et leur utilisation. ```python def one_step(position, direction) : i, j = position return {'r' : (i,j+1), 'l' : (i,j-1), 'u' : (i-1,j), 'd' : (i+1,j)}[direction] def free(position, map_rules) : return not(position in map_rules['walls']) ``` * cette version de `free` n'est pas très satisfaisante : devoir préciser la carte dans les arguments est pénible et inélégant. Le patron de conception *fabrique* offre une manière de contourner cette difficulté et d'implémenter la *fonction dépendant d'un paramètre* $free_{map} : \{positions\} \to \{True, False\}$ ```python def free_factory(map_rules) : def free(position) : return not(position in map_rules['walls']) return free ``` ### do(action, état) On peut implémenter * comme une procédure, modifiant en place l'état state. On peut renvoyer un booléen pour signifier la réussite de l'opération (c.à.d. la satisfaction des préconditions) ```python def do_inplace(action, state) : X0 = state['me'] boxes = state['boxes'] X1 = one_step(X0, action.direction) if action.verb == 'move' : if free(X1) and not (X1 in boxes) : state['me'] = X1 else : return None if action.verb == 'push' : X2 = one_step(X1, action.direction) if X1 in boxes and free(X2) and not (X2 in boxes) : state['me'] = X1 state['boxes'].add(X2) state['boxes'].remove(X1) else : return None return None ``` Le parcours correspondant: ```python state = {k : v for k,v in s0.items()} # la ligne qui précède permet de cloner s0 # pour ne pas l'écraser print(0,state) for a in 'RurrddddlDRuuuuLLLrdRDrddlLdllUUdR' : do_inplace(actions[a],state) print(a,state) print(state['boxes']==map_rules['goals']) ``` * comme une fonction retournant un nouvel état, ou None en cas d'échec ```python def do_fn(action, state) : X0 = state['me'] boxes = state['boxes'] X1 = one_step(X0, action.direction) if action.verb == 'move' : if free(X1) and not (X1 in boxes) : return {'me' : X1, 'boxes' : boxes} else : return None if action.verb == 'push' : X2 = one_step(X1, action.direction) if X1 in boxes and free(X2) and not (X2 in boxes) : return {'boxes' : {X2} | boxes - {X1} , 'me' : X1} else : return None return None ``` Le parcours correspondant: ```python state = s0 print(0,state) for a in 'RurrddddlDRuuuuLLLrdRDrddlLdllUUdR' : state = do_fn(actions[a],state) print(a,state) print(state['boxes']==map_rules['goals']) ``` ### Représentation hachable des états Les implémentations proposées ne permettent pas de stocker les états parcourus dans un dictionnaire : ```python= state = s0 print(0,state) save = {} for a in 'RurrddddlDRuuuuLLLrdRDrddlLdllUUdR' : newstate = do_fn(a,state) save[state] = newstate state = newstate print(a,state) print(state['boxes']==rules['goals']) ``` L'exécution du code précédent produit une erreur `TypeError: unhashable type: 'dict'` En effet, il n'est pas possible d'utiliser une clé *mutable* dans un dictionnaire. On peut envisager plusieurs solutions à ce problème : 1. utiliser la programmation orientée objet pour représenter les états et fournir une méthode `__hash()` aux objets de la classe `Etat` 2. représenter les états par un type non-mutable ayant une interface similaire : multiplets (`tuple`) pour les listes, ensembles figés (`frozenset`) pour les ensembles... ou multiplets nommés (`collections.namedtuple`) pour les dictionnaires (ou les objets) ```python from collections import namedtuple State = namedtuple('state',('me','boxes')) ``` 3. utiliser le transtypage à la volée pour utiliser ces types immuables lors du stockage tout en continuant à employer le code précédent Une implémentation du parcours précédent mettant en oeuvre la stratégie "sale et rapide" n°3 consiste à remplacer la ligne 1 par `state = State(**s0)` (dictionnaire vers namedtuple) et la ligne 5 par `newstate = State(**do_fn(a,state._asdict()))` (aller-retour) ## Algorithmes de recherche Pour implémenter les algorithmes de recherche non informée, on peut utiliser la fonction `search_with_parent` vue en cours. ```python def search_with_parent(s0, goals, succ, remove, insert, debug=True) : l = [s0] save = {s0: None} s = s0 while l: if debug: print("l =", l) s, l = remove(l) for s2,a in succ(s).items(): if not s2 in save: save[s2] = (s,a) if goals(s2): return s2, save insert(s2, l) return None, save ``` On utilise l'implémentation donnée en cours, légèrement altérée * pour effectuer une recherche en largeur, les états sont insérés en queue de file et retirés en tête de file * la fonction `succ` renvoie un dictionnaire avec pour clé les états suivants, et pour valeurs les actions correspondantes. Elle est fournie par une *fabrique* * la reconstruction du chemin `dict2path` est adaptée pour construire une liste de couples (état, action) * le but est implémenté à l'aide d'un prédicat (une fonction python à valeurs booléennes) * on teste si un état valide le but lors de l'insertion plutôt que lors du retrait ```python def insert_tail(s, l): l.append(s) return l def remove_head(l): return l.pop(0), l def remove_tail(l): return l.pop(), l def dict2path(s, d): l = [(s,None)] while not d[s] is None: parent, a = d[s] l.append((parent,a)) s = parent l.reverse() return l def goal_factory(rules) : def goals(state) : return state.boxes == rules['goals'] return goals def succ_factory(rules) : def succ(state) : l = [(do_fn(a,state._asdict()),a) for a in actions] return {State(**x) : a for x,a in l if x} return succ ``` Le script suivant effectue la recherche en largeur d'abord. **Il s'exécute en une trentaine de secondes sur un portable haut de gamme** ```python s_end, save = search_with_parent(s0, mygoals, mysucc, remove_head, insert_tail, debug=False) plan = ''.join([a for s,a in dict2path(s_end,save) if a]) ``` On trouve un plan de longueur 34, presque identique à celui mis en oeuvre dans le gif, après avoir visité 481 975 états intermédiaires.
